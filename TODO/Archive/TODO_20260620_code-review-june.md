## Summary

The package is in good structural health: type-annotation coverage is excellent (ruff `ANN` finds only 6 real gaps in shipped code, the rest live in a marimo notebook), there are zero unused imports/variables/redefinitions, and the converter layer already uses a clean Strategy split. The dominant theme is **duplication concentrated in the `params/` vendor parsers**: a copy-pasted source-loader, a stringly-typed mass-tolerance round-trip, and three long mixed-abstraction `extract_params` functions. The second theme is **structural typing debt** — several `dict`/`tuple`/`list` blobs are used as ad-hoc structs and a hand-maintained field tuple shadows the pydantic model. None of these are correctness bugs today; they are decay vectors that make adding the next vendor riskier than it should be.

## Top Issues

1. **Copy-pasted source-loader across ~8 vendor param modules** (major, `src/anndata_proteomics/params/diann.py:226-234` + spectronaut:119-127, peaks:70-78, fragpipe:174-185, alphapept:64, wombat:71, sage:99, metamorpheus:59-95)
   The same "source may be a path, a text file-like, or a bytes file-like" branching (`hasattr(source,'read')` / decode bytes / `Path(source).read_text()`) is re-implemented ~10 times, with subtle drift (some `seek`, some `strip`, some `decode('replace')`, sage uses `getvalue`). Source acquisition is not a parsing concern — only the parse step legitimately varies per vendor. **Why it matters:** every loader diverges independently, so an encoding/EOF bug must be fixed N times and a new vendor copies whichever neighbor was open, propagating its bug. **Fix:** `params/_common.py` already exists (it holds `format_tolerance_range`); add `read_text(source) -> str` and `read_lines(source, *, strip=False) -> list[str]` there, then replace the `_load_*` duplicates. YAML/JSON vendors call `read_text` then `yaml.safe_load`/`json.loads`. No base class. *Flagged by: GOF-001, ANTI-001 (converged).*

2. **Stringly-typed mass-tolerance round-trip** (major, `src/anndata_proteomics/params/diann.py:293-300` + model.py)
   ~8 vendor parsers build bracketed strings (`f'[-{frag} ppm, {frag} ppm]'`) that `MassTolerance.parse` immediately re-parses via two regexes. A typed `MassTolerance` already exists; the vendor layer serializes to string only so the model can deserialize back. **Why it matters:** the bracketed format is an undocumented implicit contract spread across every vendor module *and* the model's regexes; any tweak forces edits in producers and the consumer, and the detour discards type safety. **Fix:** let parsers that have numeric values construct `MassTolerance(...)` directly and pass the object to `Parameters`; keep `MassTolerance.parse` only for genuinely string-only sources (Spectronaut/PEAKS raw text). *Flagged by: ANTI-002.*

3. **`apply_rule` mixes three abstraction levels in 97 lines** (major, `src/anndata_proteomics/modifications/apply_rules.py:129-225`, cyclomatic ~30)
   One body holds high-level orchestration, a ~35-line per-match loop doing raw-offset token-position classification (N-term/C-term/before-residue/Anywhere) while mutating `stripped_chars`/`cursor`/`pending`, and a ~30-line resolution loop covering three `unknown_policy` outcomes plus terminal-index bookkeeping. The lowest register (offset bit-twiddling) sits beside the highest (the contract). **Why it matters:** this is the package's most complex function; every new modification quirk inflates it further and the offset arithmetic is exactly where a silent miscount would hide. **Fix:** extract `_strip_terminal_markers`, `_tokenize`, `_resolve_tokens`; `apply_rule` becomes a ~12-line strip→tokenize→resolve→render pipeline. Behavior identical. *Flagged by: FUNC-001.*

4. **`diann.extract_params` is a (now ~115-line) mutable-accumulator procedure** (major, `src/anndata_proteomics/params/diann.py:235-350`, cyclomatic ~18; grew in the 2026-06-19 merge)
   Builds a mutable `out` dict through defaults → command-line loop → in-log regex fallbacks → optional cfg block, with control flow keyed on "is this key already in `out`?". The cfg block silently overwrites earlier values, and `out['fragment_mass_tolerance']` is a raw number vs a bracketed string depending on which branch ran. **Why it matters:** the only way to know a field's final value is to read all ~100 lines in order; each new DIA-NN quirk adds another conditional mutation and double-bracketing bugs are easy to introduce. **Fix:** split into staged pure helpers `_defaults()`, `_from_cmdline()`, `_from_log_regex()`, `_from_cfg()` each returning a partial dict, merge with explicit precedence, normalize tolerance to typed `MassTolerance` once. *Flagged by: ANTI-004 + FUNC-002 (converged — same code, two vocabularies).*

5. **`fragpipe.extract_params` is a (now ~95-line) six-derivation block** (major, `src/anndata_proteomics/params/fragpipe.py:187-281`, cyclomatic ~17)
   After parsing the workflow into a Series it inlines six independent derivations before `Parameters(...)`: enzyme massaging, precursor+fragment tolerance string building, an FDR/MBR branch (DIA-NN vs phi-report), an override-charge branch, m/z from digest mass/charge, and quantification/MBR/protein-inference. **Why it matters:** six unrelated concerns share one scope, so a change to (say) FDR handling forces re-reading the whole function to confirm it doesn't touch tolerance state. **Fix:** extract pure helpers on the Series (`_resolve_enzyme`, `_tolerances`, `_fdr_and_mbr`, `_charge_range`, `_precursor_mz`, `_protein_inference`); `extract_params` shrinks to load→version→helpers→`Parameters(...)`. *Flagged by: FUNC-003.*

6. **Adding a vendor is Shotgun Surgery; registry already drifting** (minor→major by convergence with dead-code, `src/anndata_proteomics/params/registry.py:8-36`)
   Supporting vendor X requires coordinated edits across `params/<vendor>.py`, `registry.py`, `test_data._PROTEOBENCH_PARAM_FIXTURES`, and `parsing_rules/<vendor>/` TOML. The registry lists 11 parsers but only 7 vendors have packaged conversion rules, and the fixtures list a third subset — the surfaces are already out of sync. **Why it matters:** "support vendor X" is not localized, so a parser can exist with no rule and a rule with no fixture; omissions stay silent until a runtime `KeyError` or a skipped report row. **Fix:** drive from a single declarative table (vendor → parser → rule dir → fixture) or discover by convention; at minimum add a consistency test that fails when the three surfaces disagree. *Flagged by: ANTI-007.*

7. **Per-vendor modification-mapping helpers duplicated across all 10 vendors** (major, introduced by the 2026-06-19 ProteoBench merge)
   Every vendor parser now carries its own modification-token → ProForma-string mapper, in three recurring mechanics: (a) a `{name} (residue)` / `{name} on {residue}` text homogenizer — `_homogenize_mod`/`_homogenize_mods` in `maxquant.py:25-60`, `spectronaut.py:31-66`, `metamorpheus.py:22-56`, plus the X!Tandem variant `wombat.py:13-34`; (b) a numeric mass-tolerance lookup — `_lookup_mod_name` + a `*MASS_TO_MOD*` dict in `fragpipe.py:27-49` and `sage.py:13-29`; (c) a flat `MAP.get(mod, mod)` apply — `diann.py:348`, `peaks.py:127`, `alphapept.py:49`, `msaid.py:20-25`. The Spectronaut helper was copied verbatim from MaxQuant to dodge a cross-vendor import. **Why it matters:** identical decay vector to the source-loader (Issue 1) — the *mapping data* is legitimately per-vendor, but the shared *mechanics* now drift independently across ~6 files. **Fix:** mirror the source-loader resolution — extract the homogenizer mechanics into `params/_common.py` (e.g. `homogenize_paren_mods`, `lookup_mass_mod`), leaving each vendor only its mapping dict. Free functions, no hierarchy. Pairs with Action 1. *Flagged by: post-merge review.*

## Agreed Recommendations

- **Extract a shared source-loader into the existing `params/_common.py`** — the single highest-convergence item (GOF + ANTI). A helper, not a hierarchy.
- **Stop round-tripping mass tolerances through bracketed strings** where the parser already has numeric values (ANTI); use the typed `MassTolerance` model that already exists.
- **Break up the long `extract_params` procedures** (DIA-NN, FragPipe, MaxQuant) into staged pure helpers; FUNC and ANTI independently land on the same DIA-NN body.
- **Derive serialization field lists from the pydantic model, not hand-copied tuples** (`_SERIES_FIELDS`) (ANTI + TYPE).
- **Prefer `dict[str, object]` over `Any` / bare `dict`** at cross-module boundaries so callers must narrow (TYPE).

## Dissenting Opinions

**1. Template Method / shared base class across the ~12 vendor param modules.**
The GoF specialist explicitly judged that introducing a Template Method or shared base class spanning the vendor parsers would be a *forced/misapplied* pattern: the per-vendor parse step is the genuine variation point, the populations are tiny, and a class hierarchy would add ceremony without removing real duplication. The ANTI and GoF specialists nonetheless agree the *source-loader* duplication should be extracted. **These are compatible, not contradictory.** My call: **extract a shared free-function helper (`read_text`/`read_lines` in `_common.py`); do NOT introduce a class hierarchy.** The duplication that hurts is source acquisition (mechanical, identical), not parsing (intentionally divergent) — so factor the former and leave the latter as flat per-vendor functions.

**2. `_compute_column` dispatch (GOF-002) — function-table vs leave-as-is.**
GoF flags the if/elif-on-`column.how` chain as dispatch knowledge split across two files and suggests a dict-of-callables. The population is only 3 modes and no other specialist flagged it. **My call: low priority.** Do it opportunistically if `assemble.py` is touched for another reason; it is not worth a standalone commit. A `dict[ColumnComputeMode, Callable]` is the right shape *if* done — never a class hierarchy for 3 modes.

**3. `to_series`/`from_series` — delete vs keep (ANTI-009).**
ANTI frames these as a Boat Anchor "not called in src/." The verified dead-code ground truth contradicts the deadness claim: ~10 `test_params_*.py` modules use them as the comparison mechanism, so they are **not dead**. **My call: keep the functions, fix the real defect** — `_SERIES_FIELDS` is a hand-maintained ~28-field tuple duplicating `Parameters.model_fields` that silently goes stale when a field is added/renamed. Derive the ordered field list from `cls.model_fields` (with an explicit allow/deny list). This reconciles ANTI-009 with TYPE-007 and the dead-code finding.

**Decision — `TypedDict` vs pydantic input models for the structured-document vendors (Sage / AlphaPept / Wombat / MetaMorpheus):** the nested-config findings TYPE-002/004/005 can be expressed two ways. Option (b) costs more API surface but catches malformed vendor files at load time (valuable since these parsers must round-trip ProteoBench golden outputs) and lets tolerance be a typed field, partially absorbing the ANTI-002 string round-trip. Default to (a) `TypedDict` unless runtime guardrails on vendor input are explicitly wanted.
